[문서명] 음주운전 방지장치 통합운영관리시스템
프로세스 정의서 (Process Definition Script)

본 문서는 실제 구현된 백엔드 마이크로서비스 아키텍처(Auth Service, Company Service, Reservation Service 등)를 기반으로 작성되었으며, 각 프로세스마다 실제 API 엔드포인트와 데이터 흐름을 명시합니다.

1. 개요

  • [cite_start]프로젝트명: 음주운전 방지장치 통합운영관리시스템 구축 사업 [cite: 14]
  • 문서 목적: 본 문서는 시스템의 3대 주요 행위자인 관리자, 업체, 사용자 간의 상호작용 및 핵심 업무 절차(스크립트)를 정의합니다. 이 문서는 시스템 개발을 위한 명확한 워크플로우를 제공하는 것을 목적으로 합니다.
  • 주요 행위자 (Actors):
    • [cite_start]관리자 (Administrator): 도로교통공단(주관기관) 소속[cite: 2]. 시스템 총괄 운영자.
    • [cite_start]업체 (Operator): 공단의 인증을 받은 장치 제조, 설치 및 수리 담당자[cite: 76, 165].
    • [cite_start]사용자 (User): 음주운전 방지장치 부착 의무가 있는 운전자(대상자)[cite: 70].

2. 핵심 프로세스 정의 (Process Scripts)

[P-01] 업체 등록 및 관리 프로세스

관리자가 신뢰할 수 있는 업체를 시스템에 등록, 승인, 관리하는 절차

목적: 관리자가 신뢰할 수 있는 업체를 시스템에 등록, 승인, 관리하는 절차.
관련 행위자: 관리자, 업체

절차 (Script):

  1. [cite_start]관리자오프라인 업체로부터 참여 신청 서류(사업자등록증, 직인 사본 등)를 접수합니다[cite: 165].
  2. 관리자 관리자 시스템 > 업체 관리 > 신규 업체 등록 화면에 접속합니다.
  3. 관리자 업체의 기본 정보(업체명, 주소, 연락처 등)와 증빙 서류를 입력하고, 해당 업체의 업체 포털 접속 계정(초기 ID/PW)을 생성하여 저장합니다.
    API: POST /api/v1/admin/companies
    서비스: Company Service → PostgreSQL
    생성 데이터: Company 엔티티 (상태: PENDING), 업체 계정 (Auth Service)
  4. 관리자 등록된 업체의 상태를 검토 후 '승인(APPROVED)'으로 변경합니다.
    [데이터 흐름] '승인' 상태가 된 업체만 사용자 포털 > 설치 업체 조회 목록에 노출됩니다.
    API: PATCH /api/v1/admin/companies/{id}/approve
    상태 변경: PENDING → APPROVED
    결과: 승인된 업체만 공개 Operator 목록에 표시
  5. [cite_start]업체 관리자로부터 전달받은 계정 정보로 업체 포털 > 로그인 화면을 통해 시스템에 최초 접속하고, 필요시 비밀번호를 변경합니다[cite: 219].
    API: POST /api/v1/auth/login
    서비스: Auth Service (JWT 인증)
    역할: ROLE_COMPANY

[P-02] 장치 설치 예약 및 등록 프로세스

사용자가 장치 설치를 예약하고, 업체가 설치 완료 후 해당 장치 정보를 시스템에 등록하는 핵심 절차

목적: 사용자가 장치 설치를 예약하고, 업체가 설치 완료 후 해당 장치 정보를 시스템에 등록하는 핵심 절차.
관련 행위자: 사용자, 업체, 관리자

절차 (Script):

  1. [cite_start]사용자 사용자 포털 > 로그인 후, 설치 업체 조회 화면에서 P-01에서 '승인'된 업체 목록을 확인합니다[cite: 212].
    API: GET /api/v1/public/operators
    서비스: Company Service (Operator 조회)
    필터: status = APPROVED인 업체만 반환
  2. [cite_start]사용자 사용자 포털 > 예약 관리 > 예약 신청하기 화면에서 업체, 서비스 종류(신규 설치), 희망 일시를 선택하여 예약을 신청합니다[cite: 215].
    API: POST /api/v1/reservations
    서비스: Reservation Service → PostgreSQL
    생성 데이터: Reservation 엔티티
    초기 상태: PENDING (예약 대기)
  3. 업체 업체 포털 > 예약 관리 > 신규 예약 접수 화면에서 사용자의 예약 요청을 확인하고, 일정을 조율하여 '예약 확정' 처리를 합니다.
    [데이터 흐름] '예약 확정' 상태가 사용자 포털 > 나의 예약 내역에 실시간으로 반영됩니다.
    API: PATCH /api/v1/reservations/{id}/confirm
    상태 변경: PENDING → CONFIRMED
    결과: 사용자에게 예약 확정 알림
  4. 사용자오프라인 예약일에 업체를 방문하여 장치를 설치받습니다.
  5. 업체 (설치 완료 후) 업체 포털 > 고객/장치 관리 > 장치 설치 등록 화면에 접속합니다.
  6. [cite_start]업체 해당 사용자를 조회한 뒤, 설치한 장치의 고유 정보(시리얼 번호, 모델명 등)를 입력하여 '장치-사용자' 매핑을 완료합니다[cite: 219].
    API: POST /api/v1/company/devices/install
    서비스: Company Service
    생성 데이터:
    • ManagedDevice 엔티티 (장치 정보 + 고객 매핑, 상태: ACTIVE)
    • ServiceRecord 엔티티 (서비스 타입: INSTALLATION, 작업 일시, 담당자)
    • Customer 엔티티 업데이트 (장치 연결)
    데이터베이스: PostgreSQL (Company 서비스)
  7. [cite_start]사용자 사용자 포털 > 마이페이지 또는 설치 내역 조회 화면에서 자신의 계정에 장치 정보가 정상적으로 등록되었는지 확인합니다[cite: 211, 213].
    API: GET /api/v1/user/devices
    서비스: Company Service (사용자별 장치 조회)
    반환 데이터: 장치 정보, 설치 일시, 업체 정보
  8. [cite_start]관리자 관리자 시스템 > 대상자 관리 화면에서 해당 사용자의 상세 정보에 장치가 정상 매핑되었는지, 설치가 완료되었는지 모니터링합니다[cite: 158].
    API: GET /api/v1/admin/subjects/{id}
    서비스: Auth Service + Company Service (통합 조회)
    확인 사항: 장치 매핑 여부, 설치 완료 상태, 서비스 기록

[P-03] 운행기록(로그) 제출 및 관리 프로세스

사용자가 주기적으로 운행기록을 제출하고, 관리자가 이를 확인 및 분석하는 사후 관리 절차

목적: 사용자가 주기적으로 운행기록을 제출하고, 관리자가 이를 확인 및 분석하는 사후 관리 절차.
관련 행위자: 사용자, 관리자

절차 (Script):

  1. 사용자주기적 장치에서 추출한 운행기록(로그) 파일을 준비합니다.
  2. [cite_start]사용자 사용자 포털 > 운행기록(로그) 관리 > 로그 제출하기 화면에서 파일을 업로드하여 제출합니다[cite: 214].
    API: POST /api/v1/user/logs/submit
    서비스: Log Service (향후 구현 예정) 또는 Company Service
    저장소:
    • PostgreSQL: 로그 메타데이터 (제출 일시, 사용자, 상태)
    • File Storage: 실제 로그 파일 (CSV, TXT 등)
    처리 상태: SUBMITTED → UNDER_REVIEW → APPROVED
  3. 사용자 제출 내역 조회 탭에서 자신의 제출 이력과 처리 상태를 확인합니다.
    API: GET /api/v1/user/logs
    서비스: Log Service
    반환 데이터: 제출 일시, 파일명, 처리 상태, 관리자 코멘트
  4. [cite_start]관리자 관리자 시스템 > 장치/로그 관리 화면에서 사용자가 제출한 로그를 조회합니다[cite: 161].
    API: GET /api/v1/admin/logs
    서비스: Log Service (전체 로그 조회)
    필터: 제출 일시, 사용자, 상태별 검색
  5. [cite_start]관리자 (필요시) 로그 데이터를 분석하여 이상 징후(음주 시도 등)를 확인하고, 경찰청 TCS 연계 등 후속 조치를 준비합니다[cite: 162].
    외부 연동: TCS Mock Service (면허 검증, 위반 기록 조회)
    API: POST /api/v1/tcs/license/verify
    분석 내용:
    • 음주 측정 시도 횟수 및 결과
    • 비정상 패턴 감지 (우회 시도, 측정 거부 등)
    • 정기 제출 주기 준수 여부
    후속 조치: 경고, 제재, 재교육 등

[P-04] 장치 검·교정 및 유지보수 프로세스

법적 주기에 따른 정기 검·교정 또는 A/S 발생 시, 사용자가 예약을 하고 업체가 그 결과를 시스템에 기록하는 절차

[cite_start]목적: 법적 주기에 따른 정기 검·교정 또는 A/S 발생 시, 사용자가 예약을 하고 업체가 그 결과를 시스템에 기록하는 절차[cite: 52, 108].
관련 행위자: 사용자, 업체, 관리자

절차 (Script):

  1. [cite_start]사용자정기 또는 비정기 사용자 포털 > 예약 관리 > 예약 신청하기 화면에서 서비스 종류(검·교정 또는 A/S)를 선택하여 예약을 신청합니다[cite: 215].
    API: POST /api/v1/reservations
    서비스 타입:
    • CALIBRATION (정기 검·교정)
    • REPAIR (고장 수리/A/S)
    상태: PENDING (예약 대기)
    서비스: Reservation Service
  2. 업체 (P-02의 3, 4번 절차와 유사) 예약을 확정하고, 방문한 사용자에게 서비스를 제공합니다.
    API: PATCH /api/v1/reservations/{id}/confirm
    상태 변경: PENDING → CONFIRMED
    오프라인: 실제 검·교정 또는 수리 작업 수행
  3. [cite_start]업체 (완료 후) 업체 포털 > 고객/장치 관리 > 검·교정 결과 등록 (또는 수리 내역 등록) 화면에서 작업 결과를 입력하고 저장합니다[cite: 219].
    API: POST /api/v1/company/service-records
    서비스: Company Service (ServiceRecord 생성)
    저장 데이터:
    • 서비스 타입: CALIBRATION 또는 REPAIR
    • 작업 일시 (serviceDate)
    • 작업 내용 (description)
    • 담당자 정보 (technician)
    • 교체 부품 (있는 경우)
    • 다음 검·교정 예정일 (nextCalibrationDate)
    예약 상태 업데이트: CONFIRMED → COMPLETED
  4. [cite_start]관리자 관리자 시스템 > 장치 관리 또는 대상자 관리 화면에서 사용자의 검·교정 이행 여부와 업체가 등록한 내역을 모니터링합니다[cite: 162].
    API: GET /api/v1/admin/service-records
    서비스: Company Service (서비스 기록 조회)
    모니터링 항목:
    • 정기 검·교정 주기 준수 여부 (법정 주기 대비)
    • A/S 빈도 분석 (고장 다발 장치 파악)
    • 업체별 서비스 품질 평가
    • 미이행자 관리 (검·교정 기한 초과자)
    후속 조치: 미이행자 알림, 업체 평가 반영

3. 시스템 아키텍처 참조

마이크로서비스 구성:

  • Auth Service: 사용자 인증 및 권한 관리 (Spring Boot + Spring Security + JWT)
  • Company Service: 업체, 고객, 장치, 서비스 기록 관리 (Spring Boot + JPA + PostgreSQL)
  • Reservation Service: 설치/수리 예약 관리 (Spring Boot + JPA + PostgreSQL)
  • Log Service: 운행기록 제출 및 분석 (향후 구현 예정)
  • TCS Mock Service: 경찰청 TCS API 모의 서버 (Spring Boot + H2 Database)

인프라 서비스:

  • Eureka Server: Service Discovery & Registry
  • Config Service: 중앙 설정 관리 (Git Repository)
  • API Gateway: 라우팅, 인증, 로드 밸런싱 (Spring Cloud Gateway)

데이터베이스:

  • PostgreSQL: 사용자, 업체, 예약, 장치, 서비스 기록 등 모든 비즈니스 데이터
  • File Storage: 로그 파일, 증빙 서류 등 업로드 파일

본 문서는 실제 구현된 백엔드 API를 기반으로 작성되었으며, 모든 API 엔드포인트는 /api/v1 프리픽스를 사용합니다.
